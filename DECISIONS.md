# Decision Register

These choices were resolved before drafting final policy pages. Suggested defaults are kept as context for future policy changes, not as separate authority.

Resolved decisions are policy for drafted pages. New policy questions should be added here before changing policy pages.

| ID | Decision | Options | Suggested default | Resolution |
| --- | --- | --- | --- | --- |
| D1 | Overall style bias | OO-leaning, functional-heavy, or neutral | OO-leaning Rust: structs with methods and encapsulation by default, functional style when it is clearer | Resolved: OO-leaning Rust. Prefer structs with methods and encapsulation by default; use functional style when clearer. |
| D2 | Library vs application split | One policy for all code, or separate policies | Separate policies. Libraries are stricter; applications can be more pragmatic | Resolved: Use separate policies. Libraries are stricter about public API, errors, dependencies, and runtime assumptions; applications and CLIs can be more pragmatic. |
| D3 | Edition and MSRV | Rust 2021, Rust 2024, latest stable only, explicit MSRV | Rust 2024 unless constrained; always declare `rust-version` for crates | Resolved: Use Rust 2024 by default with explicit `rust-version`; new Rust 2024 crates default to `rust-version = "1.85"` unless project constraints require otherwise. |
| D4 | Error strategy for libraries | `thiserror` enums, hand-written structs, `snafu`, erased errors | Typed public errors, usually `thiserror`; revisit canonical struct style for public libraries | Resolved: Use typed public errors, usually with `thiserror`; hand-write impls only when avoiding a dependency or needing tighter public API control. |
| D5 | Error strategy for applications | `anyhow`, `eyre`, `miette`, domain errors everywhere | `anyhow` for internal apps/CLIs; consider `miette` for rich user-facing diagnostics | Resolved: Use `anyhow` by default for internal apps and CLIs; use `miette` for rich user-facing CLI diagnostics; keep typed domain errors where code branches on failures. |
| D6 | Panic and unwrap policy | Ban broadly, allow as assertion, allow in tests only | `Result` by default; `expect` allowed only with invariant-focused messages | Resolved: Use `Result` for recoverable failures. Panic only for violated invariants, impossible states, or test/setup failures where continuing would be a bug. Prefer `expect` with invariant-focused messages; ban bare `unwrap` outside tests (`clippy::unwrap_used` is denied, with `allow-unwrap-in-tests` permitting it in tests). |
| D7 | Polymorphism default | Enum-first, trait-first, `dyn Trait`-heavy | Closed sets use enums; open extension points use traits; `dyn Trait` for runtime heterogeneity | Resolved: Closed sets use enums; open extension points use traits; use generics for static dispatch and `dyn Trait` for runtime heterogeneity. |
| D8 | OO composition style | Method forwarding manually, delegation crates, avoid forwarding | Prefer direct composition and small traits; use delegation only when it removes real boilerplate | Resolved: Prefer direct composition and small traits. Manual forwarding is acceptable when the outer type adds a real boundary; heavy pass-through is a redesign signal. |
| D9 | Derive policy | Derive common traits eagerly, or keep impl surface minimal | Derive standard traits when semantics are obvious; hand-write `Display` | Resolved: Derive standard traits when semantics are obvious; hand-write `Display`; avoid deriving semantics-heavy traits by habit. |
| D10 | Clone policy | Clone liberally, borrow aggressively, context-dependent | Borrow at API boundaries; clone internally when it simplifies code and cost is small | Resolved: Prefer borrowed parameters and owned values at boundaries. Plain accessors borrow by default; owned snapshots use names that signal ownership. Clone freely when it keeps APIs simple or avoids lifetime complexity; optimize clones only when cost is known to matter. |
| D11 | Parameter flexibility | Concrete refs only, generic `AsRef`/`Into`, or mixed | Prefer `&str`, `&[T]`, and concrete refs; use generics when caller flexibility is valuable | Resolved: Use a mixed conservative policy. Prefer concrete refs like `&str`, `&[T]`, and `&Path`; use `impl AsRef`, `impl Into`, or generics only when caller flexibility clearly helps. |
| D12 | Newtype aggressiveness | Wrap most primitives, only wrap domain boundaries | Use newtypes for IDs, units, validated values, and public API meaning | Resolved: Use newtypes for IDs, units, validated values, and public API meaning. Do not wrap primitives aggressively when the wrapper adds no type safety, validation, or API clarity. |
| D13 | Builder style | Non-consuming builders, consuming builders, typestate builders | Use simple constructors first; use builders for many optional fields; typestate only for important invariants | Resolved: Use simple constructors first. Use `new` and `try_new` for required fields, builders for many optional fields, and typestate only for important invariants. |
| D14 | Visibility style | Private by default, `pub(crate)` common, public fields common | Private by default; `pub(crate)` for internal seams; public fields only for plain data | Resolved: Private by default. Use `pub(crate)` for internal boundaries; use public fields only for plain data with no invariants. |
| D15 | Re-export/prelude policy | Public facade, explicit deep modules, public prelude | Use focused public facades; prelude only for broad library ecosystems | Resolved: Use focused public facades and avoid broad preludes by default. Add a prelude only for broad ecosystem crates where users commonly need many traits or types together. |
| D16 | Async runtime | Tokio-first, runtime-agnostic, smol/minimal | Tokio-first for apps/services; runtime-agnostic surfaces for reusable libraries | Resolved: Tokio-first everywhere. Libraries may expose Tokio-based APIs when async behavior is central, but must not silently install a runtime or hide task lifecycle from callers. |
| D17 | Async task style | Free spawning, structured lifecycle, mixed | Prefer owned task lifecycle and explicit shutdown paths | Resolved: Use structured task lifecycle. Applications and services that spawn tasks should keep explicit owners, named handles, cancellation/shutdown paths, and join during teardown. Reusable libraries should avoid hidden spawning by exposing futures/streams, or return an owner type when the library truly owns the task. Avoid fire-and-forget `tokio::spawn` except for truly disposable best-effort work. |
| D18 | Shared state style | Channels first, `Arc<Mutex<T>>` first, mixed | Choose the simpler model; channels for ownership transfer, locks for shared state | Resolved: Choose the simplest model. Prefer ownership first, use `Arc` for shared ownership, `Mutex` or `RwLock` for real shared mutable state, and channels for ownership transfer. |
| D19 | Logging stack | `tracing`, `log`, mixed | `tracing` for apps/services; `log` facade for reusable libraries | Resolved: Use `tracing` everywhere. Libraries may emit spans and events, but applications own subscriber setup and filtering. |
| D20 | Clippy strictness | Minimal, curated strictness, blanket pedantic | Curated strictness; do not deny all pedantic lints by default | Resolved: Use curated strictness. Use Fabro's workspace lint rules as a starting point, but tailor custom denies and project-specific lints per project. |
| D21 | Lint overrides | `allow`, `expect`, source-local only, Cargo lints | Prefer `#[expect]` with a reason; keep project-wide lints in Cargo when practical | Resolved: Prefer local `#[expect(..., reason = "...")]` for justified exceptions; keep shared lint policy in `Cargo.toml`; disable broadly rejected noisy lints centrally. |
| D22 | Iterator vs loop style | Iterator chains by default, loops by default, mixed | Use the clearer form; avoid clever chains when a loop communicates intent better | Resolved: Use the clearer form. Prefer iterator chains for simple transforms and loops for branching, mutation, early exits, or multi-step logic. Clippy is authoritative for local iterator-vs-loop idioms; refactor to satisfy it instead of adding line-level lint bypasses. |
| D23 | Testing posture | Unit-heavy, integration-heavy, balanced | Balanced behavior-focused testing: unit tests near focused logic, integration tests for public behavior/workflows, skip doctests by default | Resolved: Use balanced behavior-focused testing. Put unit tests near focused logic, integration tests around public behavior and workflows, and skip doctests by default. Add doctests only when a project explicitly opts into maintaining public rustdoc examples. |
| D24 | Test tooling | Plain `cargo test`, `nextest`, `proptest`, `insta`, `criterion` | Nextest-first for normal tests; add `insta` when snapshots are valuable; add property and benchmark tools only when the code benefits | Resolved: Use `cargo nextest run` as the default local and CI test runner. Add `insta` when snapshot tests make complex output easier to review. Add `proptest` for broad input invariants and `criterion` for real performance-sensitive code; prefer `proptest` over `quickcheck` for new property tests. |
| D25 | Unsafe and macros | Ban, allow with review, broad tolerance | Ban unsafe by default; allow `macro_rules!` and proc macros only when strongly motivated by code simplicity | Resolved: Ban project-written unsafe code by default with `unsafe_code = "deny"`. Allow `macro_rules!` and proc macros when there is a strong motivation to improve code simplicity, remove real repetition, or encode a clear local pattern; avoid macros for cleverness or minor convenience. |
| D26 | Formatting policy | rustfmt defaults, small curated config, or strong custom formatting | Use rustfmt defaults unless the project checks in a config | Resolved: Use pinned nightly rustfmt with a checked-in `rustfmt.toml`; reuse the Fabro settings and run `cargo +nightly-2026-04-14 fmt --check --all` in CI. |
| D27 | Dependency policy | Conservative, pragmatic, or lean/minimal | Conservative for libraries; pragmatic for applications | Resolved: Use conservative dependency policy for libraries and pragmatic dependency policy for applications. Add dependencies only when they remove real complexity, provide mature domain behavior, or materially improve application clarity. |
| D28 | Cargo feature defaults | Additive minimal defaults, batteries-included defaults, or no feature flags unless forced | Additive minimal defaults for libraries; direct dependencies for applications | Resolved: Use additive features with minimal defaults for reusable libraries. Use `default = ["std"]` only when `no_std` support is real; otherwise omit feature flags until there is an actual optional integration. Applications may depend directly on what they use. |
| D29 | MSRV impact from dependencies | MSRV-aware by default, library-only MSRV checks, or current-stable-first | MSRV-aware by default | Resolved: Check dependency and API changes against declared MSRV by default. If a change raises practical MSRV, choose a compatible version or deliberately update `rust-version`. |
| D30 | Acronym casing | Rust-style acronyms, all-caps acronyms inside names, or project-specific per acronym | Rust-style acronyms | Resolved: Use Rust-style acronym casing in names, such as `HttpClient`, `UrlParser`, `JsonBody`, and `ApiToken`; keep all-caps only for constants and conventional initialisms that are fully uppercase by Rust convention. |
| D31 | Import style | Explicit module-level imports, mostly fully qualified paths, or broad/glob imports | Explicit module-level imports grouped by rustfmt | Resolved: Use explicit module-level imports grouped by rustfmt. Warn on wildcard imports; allow globs mainly in tests, test support, or intentional prelude-style APIs. |
| D32 | Getter and public field policy | Getter-heavy, public-field-heavy, or private fields with selective accessors | Private fields with selective Rust-style accessors | Resolved: Use private fields by default, public fields for plain data with no invariants, and selective Rust-style accessors when callers need read access. Plain accessors borrow unless returning a small `Copy` value. Avoid generating getters for every field by habit. |
| D33 | Public documentation coverage | Document every public item, document public surfaces, or document only non-obvious public items | Document public surfaces, stricter for libraries | Resolved: Document only non-obvious public items. Do not require `#![deny(missing_docs)]`; add docs when names and signatures are not enough, behavior has caveats, or a public type/module defines an important concept. |
| D34 | Rustdoc example error handling | Prefer `?`, allow `unwrap`, or use `expect` with messages | Prefer fallible examples with `?` | Resolved: Prefer fallible rustdoc examples using `?`; avoid bare `unwrap` in public docs. Use `expect` only for guaranteed fixture/setup invariants when `?` would obscure the example. |
| D35 | Trait design policy | Small behavior-focused traits, open public traits, trait-light, or broad capability traits | Small behavior-focused traits; public traits are open only when external impls are intended; use sealed traits for controlled extension | Resolved: Use small, behavior-focused traits. Public traits are open only when external impls are intended; use sealed traits for controlled extension. |
| D36 | Typestate vs runtime state | Runtime validation/state enums by default, broad typestate for ordered workflows, avoid typestate except high-stakes APIs, or library/application split | Runtime validation/state enums by default; typestate only for important invariants or protocols where invalid ordering should not compile | Resolved: Use typestate broadly for workflows with ordered states. Keep runtime state enums for dynamic state, persistence, heterogeneous collections, and cases where compile-time states add more ceremony than clarity. |
| D37 | Public API evolution strictness | Conservative semver, moderate semver, flexible early APIs, or application-first | Conservative semver for published library-style code | Resolved: Use an application-first policy. Treat public API evolution as mostly irrelevant outside published crates or APIs consumed outside the repo; optimize for simplicity and accept breaking changes in internal application code. |
| D38 | Clone call style | `Rc::clone`/`Arc::clone` for shared pointers, `.clone()` everywhere, or mixed emphasis | Use `Rc::clone(&value)` and `Arc::clone(&value)` for shared pointers; use `.clone()` for ordinary values | Resolved: Use `.clone()` everywhere for consistency, including `Rc` and `Arc`. |
| D39 | Lifetime complexity tolerance | Low tolerance, medium tolerance, high tolerance, or library/application split | Low tolerance: prefer owned return values and clones; use explicit lifetimes only when borrowing is core to the API | Resolved: Use low tolerance for lifetime complexity. Prefer owned return values and clones when borrowing would complicate the API; borrow from plain accessors and use explicit lifetimes only when borrowing is core to the API. |
| D40 | Specialized collection threshold | Std-first, pragmatic app dependencies, minimal dependencies, or deterministic-first | Std-first collections; add specialized collections when semantics, deterministic ordering, or known performance needs justify them | Resolved: Use `Vec`, `HashMap`, `BTreeMap`, and other standard-library collections by default. Add specialized collection crates only when required semantics, deterministic ordering, or known performance needs justify them. |
| D41 | Error taxonomy and layer boundaries | Layered branch-oriented, fine-grained everywhere, coarse-grained, or application-erased | Layered branch-oriented errors with source-chain preservation | Resolved: Use layered, branch-oriented errors. Define typed domain errors where callers branch, convert infrastructure errors at boundaries, preserve structured data and source chains, add context at layer crossings, and render errors to strings only at external boundaries. |
| D42 | Error propagation context density | Boundary-rich inner-sparse, context at every fallible call, minimal context, or user-message-first | Boundary-rich, inner-sparse context with preserved source chains | Resolved: Add context at operation and layer boundaries, keep inner propagation sparse when typed errors already explain the local failure, preserve sources, and avoid restating lower-level errors in context messages. |
| D43 | Validation and invariant location | Parse at boundaries, validate at every use, runtime checks on plain structs, or type-level whenever possible | Parse and construct at boundaries; encode invariants in types | Resolved: Validate at input boundaries, encode invariants in newtypes and constructors, and keep internal code working with trusted types. Re-check only when state can become stale or constraints depend on changing external conditions. |
| D44 | Logging and observability style | Structured operation traces, minimal logging, verbose audit trail, or debug-first | Structured operation traces with spans, fields, and meaningful events | Resolved: Use structured operation traces. Create spans for operations, attach IDs/state as fields, emit events for meaningful milestones and errors, keep INFO sparse and high-signal, use DEBUG for investigation details, and avoid prose-only logs. |
| D45 | Sync vs async project posture | Project-specific documented choice, async for real I/O, async-first applications, sync-first unless forced, or runtime-boundary split | Project-specific documented choice; Tokio when async is chosen | Resolved: Treat sync vs async as a critical project-level architecture decision. Each project must explicitly document whether it is sync-first, Tokio-first, or mixed; where async is allowed; and which runtime/testing/shutdown conventions apply. If a project has no documented posture, establish one before broad async changes. |
| D46 | Cancellation, shutdown, and blocking work | Cooperative shutdown, abort-first shutdown, completion-first shutdown, or timeout-heavy shutdown | Cooperative shutdown with explicit cancellation, owned joins, boundary timeouts, and blocking work isolation | Resolved: Use cooperative shutdown by default. Pass explicit cancellation signals into long-lived tasks, use `select!` to race work with shutdown, join owned tasks during teardown, put timeouts at operation and external boundaries, and use `spawn_blocking` or dedicated pools for real blocking or CPU-bound work. |
| D47 | Concurrency primitive selection | Ownership-shape first, Tokio primitives throughout async code, channels-first, or lock-first | Use the simplest primitive by ownership shape | Resolved: Choose concurrency primitives by ownership shape. Prefer owned values first, use channels for ownership transfer, use standard-library locks for short synchronous critical sections, use Tokio locks only when async waiting or holding across `.await` is truly needed, and use `rayon`, `spawn_blocking`, dedicated pools, or work queues for sustained CPU or blocking work. Fabro-style lints are relevant guardrails for blocking std APIs on Tokio paths, but they are not a blanket ban on `std::sync`. |
| D48 | Control flow style | Clarity-first branching, expression-heavy concise style, match-first explicit style, or guard-clause happy path | Clarity-first branching | Resolved: Use clarity-first branching. Use `?`, `let else`, `if let`, `while let`, and `match` to make branches and exits explicit; use combinators only when they remain simple local transformations. Clippy is authoritative for local control-flow idioms; refactor to satisfy it instead of adding line-level lint bypasses. |
| D49 | Option and Result idioms | Simple combinators with explicit branching, combinator-forward style, match-first style, or error-context-first style | Simple combinators, explicit branching when behavior matters | Resolved: Use simple combinators and explicit branching when behavior matters. Use `?`, `ok_or_else`, `map`, `filter`, `transpose`, and `unwrap_or_else` for short local transforms; switch to `let else`, `if let`, or `match` when branching, side effects, context, or recovery logic matters. Clippy is authoritative for combinator-vs-branching idioms; refactor to satisfy it instead of adding line-level lint bypasses. |
